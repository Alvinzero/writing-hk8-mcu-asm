# 03 寄存器与内存使用规范

## 1. 三类空间必须分开建模

HK64S8101 当前工程基线：

| 空间 | 范围 | 容量/性质 | 正确用途 |
|---|---:|---|---|
| SFR | `0x00..0x7F` | 特殊功能寄存器 | GPIO、timer、状态等运行时硬件控制 |
| SRAM | `0x80..0xBF` | 64 bytes | 业务状态、计数器、scratch |
| OPTION | `0x00..0x1F`，独立地址域 | 烧录/配置空间 | oscillator、LVR、WDT 等芯片配置 |
| program | `0x0000..0x03FF` words | 1024 words / 2048 physical bytes | 指令、DB 表和向量 |

数值范围重叠不代表空间相同。特别是 `OPT_LVR=0x00` 与 SFR `0x00` 不能互换；OPTION 不是可被普通 `MOV R,A` 读写的运行时 RAM。

适用规则：`HK-MEM-001..007`、`HK-LAYOUT-001..002`。

## 2. SFR 名称的唯一来源

公司 IDE 交付源码必须使用 `REG825.INC` 的正式名称。GPIO 关键名称和地址：

| Port A | 地址 | Port B | 地址 | 含义按项目命名 |
|---|---:|---|---:|---|
| `PA_PPU` | `30H` | `PB_PPU` | `70H` | pull-up |
| `PA_PPD` | `31H` | `PB_PPD` | `71H` | pull-down |
| `PA_POD` | `32H` | `PB_POD` | `72H` | open-drain |
| `PA_PSL` | `33H` | `PB_PSL` | `73H` | special selection |
| `PA_POE` | `35H` | `PB_POE` | `75H` | output enable |
| `PA_PIO` | `38H` | `PB_PIO` | `78H` | pin I/O data |

正式名称/地址策略见 [rules/register-alias-policy.json](rules/register-alias-policy.json)，完整位字段和描述见 [rules/register-reference.json](rules/register-reference.json)。任何芯片、`REG825.INC` 或 register metadata 版本变化都必须重生成并交叉验证，而不是手工沿用此表。

2026-07 register metadata 包含 407 行明细、96 个聚合寄存器，并已把 8 个 GPIO 名统一为 `*_PPU/*_PPD/*_POD/*_POE`。但它与当前 `REG825.INC` 仍有 4 个显式差异：

| 类型 | 地址 | `REG825.INC` | register metadata | 当前处置 |
|---|---:|---|---|---|
| INC only | `04H` | `STATUS` | 无聚合项 | 源码继续使用 `STATUS` |
| name mismatch | `24H` | `LVD` | `LVD1` | 源码继续使用 `LVD`；位语义标 OPEN |
| metadata only | `26H` | 无 | `LVD2` | FAE/编译器确认前限制使用 |
| metadata only | `27H` | 无 | `LVD3` | FAE/编译器确认前限制使用 |

AI 和人工评审不得静默把 `LVD1` 改写为 `LVD`，也不得因 metadata 有定义就假定 company IDE 接受 `LVD2/LVD3`。

## 3. Python 内部别名不是源码 API

当前 Python assembler 模块内部使用了以下别名：

| 公司正式名 | Python 内部名 |
|---|---|
| `PA_PPU` | `PA_PU` |
| `PA_PPD` | `PA_PD` |
| `PA_POD` | `PA_OD` |
| `PA_POE` | `PA_OE` |
| `PB_PPU` | `PB_PU` |
| `PB_PPD` | `PB_PD` |
| `PB_POD` | `PB_OD` |
| `PB_POE` | `PB_OE` |

`asmc_compile.py` 会在兼容层内转换正式名。规范要求：

- 源码始终写 `PA_PPU/PB_POE` 等正式名。
- 禁止为了让 Python 模块通过而把 `PA_PU/PB_OE` 写回 ASM。
- 工具链适配只能存在于编译器 adapter，不得污染业务源码。
- 新增 alias 时要同时更新 `register-alias-policy.json`、静态检查器和基准探针。

## 4. 禁止无必要的 SFR 二次别名

不推荐：

```asm
SDA_PORT EQU PB_PIO
SDA_OE   EQU PB_POE
BCLR SDA_OE,7
```

推荐：

```asm
BCLR PB_POE,7         ; BOARD: SDA=PB7
```

原因：历史工具链对“符号解析为 SFR 后再作为某些指令操作数”的兼容性不一致；二次别名还会隐藏真实端口和寄存器类型。可读性应通过 board profile、函数名和注释获得，而不是重命名 SFR。

若未来公司 IDE 与 CLI 均通过专项探针证明某类 alias 稳定，可申请规则例外；在此之前，关键 SFR 直接使用正式名。

## 5. SRAM 分配表是交付必需品

正式 ASM 文件头必须列出全部使用的 `80H..BFH`，至少包含：

| 字段 | 含义 |
|---|---|
| address | SRAM 地址 |
| name/purpose | 业务含义 |
| class | `persistent`、`scratch`、`handoff` 或 `isr` |
| owner | 哪个模块/函数拥有 |
| lifetime | 生命周期 |
| clobbered_by | 可能覆盖它的调用链 |

示例：

```asm
; SRAM ALLOCATION
; 80H I2C_SHIFT    scratch    owner=I2C_SEND       lifetime=one call
; 81H I2C_COUNT    scratch    owner=I2C_SEND       lifetime=one call
; 82H OLED_CMD_TMP handoff    owner=OLED_CMD       lifetime=nested calls
; 88H TABLE_INDEX  scratch    owner=SEND_CHUNKx    lifetime=one chunk
; 89H TABLE_WORDS  scratch    owner=SEND_CHUNKx    lifetime=one chunk
; 90H DIGIT_1000   persistent owner=COUNTER         lifetime=main loop
; 91H DIGIT_100    persistent owner=COUNTER         lifetime=main loop
; 92H DIGIT_10     persistent owner=COUNTER         lifetime=main loop
; 93H DIGIT_1      persistent owner=COUNTER         lifetime=main loop
```

## 6. persistent 与 scratch 强制分区

项目中“`0000 -> 0101 -> 0202`”一类数码管故障的高概率根因，是长期 BCD 状态与延时、段码转换或嵌套调用 scratch 复用了同一 SRAM 地址。

强制原则：

1. 长期状态和 scratch 不得共址，即使它们在当前控制流上“看起来不会同时使用”。
2. 嵌套 `CALL` 的 scratch 生命周期覆盖整个被调用链。
3. ISR 与 main 共享地址时必须有原子性/保存恢复设计；默认禁止共享 scratch。
4. 调试 probe 可以临时分配地址，但合入正式文件前必须进入统一分配表。
5. 删除函数后同时删除其 SRAM 所有权记录，避免形成“幽灵保留区”。

推荐按项目建立区间，而不是把以下示例当全局 ABI：

| 示例区间 | 建议用途 |
|---|---|
| `80H..8FH` | 驱动短生命周期 scratch / handoff |
| `90H..9FH` | 业务 persistent state |
| `A0H..AFH` | ISR 或通信缓冲区，若实际需要 |
| `B0H..BFH` | 预留、诊断或扩展 |

这只是分区策略示例；真正分配必须由项目自身的 SRAM 表决定。

## 7. A、flags 与 SRAM clobber 契约

A 不是可自由跨调用保存的全局变量。每个函数默认：

- 可以破坏 A。
- 可以改变 flags。
- 只能写入契约列出的 SRAM。
- 不可重入。

契约示例：

```asm
; I2C_SEND
; IN:        A = transmit byte
; OUT:       80H bit7 = sampled ACK level
; CLOBBERS:  A, 80H, 81H, flags
; REENTRANT: no
I2C_SEND:
  ; ...
```

调用方若还需要原 A，必须在调用前保存到专属 handoff/persistent 地址；不得假设 `CALL` 后 A 未变。

## 8. SRAM 直接地址与业务符号

当前工具链最稳妥的兼容路径是直接地址，例如 `MOV 90H,A`。为了兼顾可读性：

- 文件头 SRAM 表必须将地址映射到业务名。
- 同一地址只允许一个 owner/lifetime 定义。
- 代码注释可写业务名：`MOV 90H,A ; DIGIT_1000`。
- 若要使用 SRAM `EQU` 符号，必须先对目标公司 IDE 和 CLI 分别做操作数探针，并把结果写入项目工具链 profile；不能默认和 SFR alias 一样可靠。

## 9. GPIO 初始化是整寄存器所有权操作

常见初始化：

```asm
MOV A,#00H
MOV PA_PPU,A
MOV PA_PPD,A
MOV PA_POD,A
MOV PA_PSL,A

MOV A,#6CH
MOV PA_POE,A
```

整寄存器写会影响同一 port 上的所有 pin。模块化代码必须声明：

- 它拥有的 pin mask。
- 初始化是否覆盖非本模块 pin。
- 与其他外设共享 port 时使用 read-modify-write 还是集中式 port manager。
- 复位默认态、输出数据和 output enable 的先后顺序，以避免毛刺。

禁止把一个“只控制 PA2”的驱动写成无说明的 `MOV PA_POE,#...` 式全口覆盖。

## 10. OPTION 配置门禁

OPTION 值必须独立保存和评审，至少记录：

- chip model/revision。
- oscillator、clock、WDT、LVR 等目标值。
- 生成工具和版本。
- 烧录前值、烧录后回读值。
- 与延时、I2C 时序、低功耗和启动行为的关联。

当前 CLI 闭环的 README 明确说明离线下载和 OPTION 字节仍是后续能力，因此不能把“程序 BIN 已 verify”解释为 OPTION 也正确。

## 11. 中断与并发

若启用中断：

- 为 ISR 单独分配 SRAM，或在入口保存、出口恢复所有共享资源。
- 主循环和 ISR 同时访问多字节状态时，定义原子区。
- ISR 禁止调用未证明可重入的 I2C/OLED/动态扫描函数。
- 明确中断嵌套策略和最大 `CALL` 深度。
- 中断向量、flag 清除和 `RETI` 行为必须绑定具体芯片资料与 E1 测试。

仅写一个 `ORG 0x0008 / RETI` 表示“当前未使用 ISR”，不是完整中断设计。

## 12. 内存审查输出

AI 或评审工具至少输出：

```json
{
  "sfr_names": ["PB_PPU", "PB_POE", "PB_PIO"],
  "internal_aliases_found": [],
  "sram_addresses": ["0x80", "0x81", "0x88", "0x89"],
  "sram_conflicts": [],
  "option_profile": "separate-artifact-or-unresolved",
  "program_range_words": ["0x0000", "0x03FF"]
}
```

静态工具可以列出地址，却不能自动理解所有生命周期。SRAM 分配冲突的最终结论必须同时审查函数调用图和 clobber 契约。

## 13. 审查清单

- [ ] 正式 SFR 名来自目标版本 `REG825.INC`。
- [ ] 无 `PA_PU/PB_OE` 等 Python 内部名。
- [ ] 已读取 `register-reference.json` 位字段，并检查 `metadata_conflicts`。
- [ ] 未擅自使用 `LVD1/LVD2/LVD3` 解决当前 OPEN。
- [ ] SFR、SRAM、OPTION、program space 没有混淆。
- [ ] 文件头列出每个 `80H..BFH` 地址。
- [ ] persistent 与 scratch 不共址。
- [ ] 嵌套调用的 scratch 生命周期无冲突。
- [ ] 每个函数声明 A、flags 和 SRAM clobbers。
- [ ] 整端口初始化不会破坏其他模块 pin。
- [ ] OPTION 有独立 profile、烧录和回读证据。
