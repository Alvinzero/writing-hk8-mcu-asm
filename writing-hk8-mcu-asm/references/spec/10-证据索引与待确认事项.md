## 4. 关键实板基线

### OLED 字符

`ssd1306_oled_heello_db_raw_verified.asm`

- SHA256：`36dc9cff310d30c08e57b9a7dec71af8e30257203abebbc383cc05ab775bc0eb`。
- 证明：原始 DB、TABL->TABH、同页小表、ACK 释放、SSD1306 page path。
- 局限：绑定 PB7/PB6、0x3C、当前 OLED 模组和初始化方向。

### OLED 日期方向

- `2026/7/24` 的标准 5x7 数字/斜杠已确认：文本和单字列序不变，每列 byte 做 bit 反转。
- `2026年8月1号` 的首次 16 像素高混合字模实板反馈为：文本块顺序正确，每个字符上下和左右均镜像。该失败证据证明 5x7 结论不能推广到多 page 字模。
- 修订流程使用 `ssd1306_page_bitmap.py` manifest，按 `8,8,8,8,16,8,16,8,16` 分块执行逐字符水平镜像和全像素行垂直镜像，并由 `hk8asm.py` 重新提取 ASM 的 192 个实际发送 byte 做 SHA256 比对。
- 局限：新的双轴修订版在用户再次烧录确认前状态仅为 `released`，不能列为 E1 正向显示基线。

### OLED 64×64 头像

`ssd1306_oled_avatar_64x64_db_raw_verified.asm`

- SHA256：`c93b6a20c58bdbefb8b71f6dfc14e2be03e9c4f6edf769420c87effe07bba1b8`。
- MAP SHA256：`fc2a34596166f2b9aff7902430b599de9c834d4e2b12270db837c7e4dfb83f6e`。
- 证明：512-byte 大表 `224+224+64` 分块、`0x00/01/02` 页内 sender、64×64 window。
- 局限：不证明其他分块大小或 simulator 跨页正确。

### 数码管

`seven_segment_1234.asm` 与 `seven_segment_counter_0001_9999.asm`：

- 固定非对称位序、混合极性、全关值和状态分区经验。
- 只绑定当前开发板短接/接线。

## 6. 指令证据

- cycles/flags 原始字段。
- metadata correction。

当前元数据与规范化状态：

## 8. 工具与规范的置信边界

| 资料 | 能证明 | 不能证明 |
|---|---|---|
| 静态检查器 | 语法模式、地址模型、部分 pair/alias/range 风险 | 电气行为、全部 SRAM 生命周期 |
| Python compile probe | 当前模块生成某 machine word | DB 交付、硬件语义 |
| company IDE MAP/BIN | 实际工件布局/编码 | 外设功能正确 |
| simulator | 模型覆盖的控制流 | 跨页表、电气层、真实 timing |
| flash verify | 镜像写入/回读一致 | 功能和 OPTION 一定正确 |
| 实板 | 当前配置下真实结果 | 自动泛化到新板/新版本 |

## 9. 待确认事项

### `OPEN-ROM-BUS`

**问题**：company IDE DB 半字节物理变换与实板 `TABL/TABH` 逻辑之间的芯片级 ROM 数据总线原因。

**Owner**：芯片设计/FAE。

**当前处置**：不阻塞已验证 raw DB 路径；禁止据此推导新的补偿格式。

**关闭证据**：官方数据通路说明 + marker 机器字解释 + HK64S825 实板真值表。

### `OPEN-RET-AK`

**问题**：`RET A,#K` 在量产硅上的确切语义、A/PC/stack/flags 行为。

**Owner**：芯片设计/FAE。

**当前处置**：正式关键项目禁用。

**关闭证据**：官方 ISA 定义 + 边界 immediate 实板 probe。

### `OPEN-CPL`

**问题**：`CPL/CPLR` 的确切按位语义、A/R 写回和 flags。

**Owner**：芯片设计/FAE。

**当前处置**：关键逻辑禁用。

**关闭证据**：`00/01/55/AA/FF` 等真值表，company IDE machine word 与实板一致。

### `OPEN-DB-ODD`

**问题**：company IDE 对单个奇数 byte `DB` 的末尾填充/配对规则。

**Owner**：compiler maintainer。

**当前处置**：每条 DB 显式偶数字节；需要时显式 pad。

**关闭证据**：奇数字节 marker 的 MAP/BIN/HEX + TABL/TABH 实板读取。

### `OPEN-REG-LVD`

**问题**：当前 `REG825.INC` 定义 `LVD@24H`，2026-07 register metadata 定义 `LVD1@24H`、`LVD2@26H`、`LVD3@27H`；同时 metadata 不含 `STATUS@04H` 聚合项。

**Owner**：芯片设计/FAE + 编译器维护者。

**当前处置**：`24H` 交付源码继续使用 `LVD`；`LVD2/LVD3` 在 INC、parser 和实板语义确认前限制使用。AI 不得自行建立别名。

**关闭证据**：更新后的正式 INC + company IDE 编译探针 + 地址/位字段官方说明 + 必要实板验证。

## 10. 新建议确认项

以下不是已知错误，但在公司推广前建议补齐：

- HK64S825 各型号 program/SRAM/vector 差异矩阵。
- hardware return stack 深度与溢出行为。
- flags 对 65 个变体的正式定义和边界测试。
- OPTION 字节的公司级 schema、默认值和烧录/回读流程。
- I2C clock stretching、bus recovery、NACK/retry 规范。
- interrupt context save/restore ABI。
- 不同 company IDE 版本的 reproducible build 能力。

AI 必须把这些作为“需要时触发的 unresolved input”，不能直接写入 active 硬规则。

## 11. 证据新增流程

新结论进入规范：

1. 创建最小、非对称、单变量 probe。
2. 保存 source/project/tool version/board profile。
3. 保存 MAP/BIN/HEX 和 SHA256。
4. 记录 simulator 结果，但不提高其等级。
5. 完成实板 expected/actual、波形/照片/日志。
6. 分配 claim ID、evidence level 和 owner。
7. 更新对应 rule 的 evidence/confidence/status。
8. 更新 version、source manifest、probe results 和 changelog。
9. 对所有受影响模板做回归。

失败假设也应保留为 deprecated evidence，防止后续 AI 重复踩坑。

## 12. 规范版本策略

建议 semantic versioning：

- MAJOR：改变已验证运行时语义、地址模型或删除兼容路径。
- MINOR：新增芯片/board profile、规则、模板或工具能力。
- PATCH：文档澄清、修复链接、增加不改变约束的证据。

每次发布记录：

```yaml
version:
date:
source_manifest_sha256:
changed_rules:
new_evidence:
deprecated_conclusions:
toolchain_versions:
revalidation_results:
```

## 13. 可重建分析快照

`tools/build_analysis_snapshot.py` 是本次仓库分析快照生成器。复制到独立目录后使用显式参数：

```powershell
python tools/build_analysis_snapshot.py `
  --repo <authorized-toolchain-repo> `
  --compiler-root <authorized-compiler-root> `
  --instruction-metadata <instruction-metadata.json> `
  --register-metadata <register-metadata.json> `
  --spec-root <spec-root>
```

脚本不得修改被分析仓库 ASM；输出仅写入指定 `spec-root/analysis` 和机器参考文件。运行前先归档当前规范，以免无意覆盖已发布快照。
