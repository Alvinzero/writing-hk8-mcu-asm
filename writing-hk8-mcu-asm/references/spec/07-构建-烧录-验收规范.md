# 07 构建、烧录与验收规范

## 1. 交付状态机

ASM 项目只能按下列状态前进：

```text
draft
  -> buildable
  -> flash_candidate
  -> hardware_verified
```

| 状态 | 必需条件 | 明确不代表 |
|---|---|---|
| `draft` | 输入/约束已记录，可供评审 | 能编译或可烧录 |
| `buildable` | 静态检查和目标编译器通过，0 error / 0 warning | 工件布局/数据正确 |
| `flash_candidate` | MAP/BIN/HEX、容量、DB marker、hash、配置均审计通过 | 功能正确 |
| `hardware_verified` | 按板级 checklist 完成 E1 验收并保存证据 | 可自动泛化到其他板卡 |

不得从 `buildable` 直接跳到 `hardware_verified`；MTP verify 只证明传输/回读，不证明程序功能。

适用规则：`HK-BUILD-001..009`。

## 2. 工具链能力矩阵

| 能力 | `company_ide` | `python_source_module_cli` | `simulator` | `hardware` |
|---|---|---|---|---|
| 普通指令生成 | 支持 | 65/65 代表探针通过 | 执行子集 | 最终语义 |
| `DB` 生成 | **支持，E2** | **不支持，BLOCKER** | 依赖 ROM 输入 | 最终读取语义 |
| MAP | 真实交付依据 | 无 DB 项目可辅助 | 不适用 | 不适用 |
| BIN | 每 word 小端；DB 有特殊物理编码 | DB 缺失 | 不适用 | 烧录/回读对象 |
| 跨页 `TABL/TABH` | 可构建 | 指令可生成 | 固定 page 0，模型错误 | 同页 sender 方案已验证 |
| 功能验收 | 否 | 否 | 低等级辅助 | **唯一 E1** |

目标工具链必须在文件头和构建记录中显式声明。禁止先随意选择工具，出错后再调整源码适配工具缺陷。

## 3. 构建前门禁

构建前执行 [checklists/pre-build.md](checklists/pre-build.md)，至少确认：

- chip、board profile、clock、OPTION、外设参数已冻结。
- 源码角色为 production/flash candidate，不是 probe/check/sanity。
- 静态规则无 BLOCKER/ERROR。
- SRAM allocation、clobbers、ORG layout 完整。
- 含 DB 时选择 company IDE。
- 已验证大表未被无意改写；byte count/hash 可比对。

命令：

```powershell
python tools/asm_static_check.py path/to/main.asm --toolchain company_ide
```

查表项目增加：

```powershell
python tools/asm_static_check.py path/to/main.asm `
  --toolchain company_ide `
  --map path/to/build/main.map `
  --table-pair TABLE0:SEND_TABLE0
```

## 4. Python CLI 使用边界

无 DB 项目可用当前 CLI 做快速编译：

```powershell
python D:/hk64s8x-cli/asmc/scripts/asmc_compile.py compile `
  --workspace D:/project `
  --source main.asm `
  --compiler-source-root D:/path/to/HK_ASM_Compiler `
  --json
```

只有以下全部满足，CLI 结果才可进入后续工件审查：

- 源码和所有 include 不含 `DB`。
- `errors=0` 且 `warnings=0`。
- chip model 与 project 配置一致。
- MAP 范围、BIN size、symbols 符合预期。
- CLI adapter 的 source manifest 与规范基线一致，或已重跑探针。

一旦源码含 `DB`，必须立即触发 `HK-TOOLCHAIN-DB-001`，不得因 CLI 的 JSON status=success 继续烧录。

## 5. company IDE 构建要求

含 DB 或准备公司正式交付时：

1. 使用目标版本 company IDE 打开 `.hkproj`。
2. 核对 `mcu_type`、source file、build path、warning level、output format。
3. clean/build 的实际操作按公司 IDE 流程执行；不得沿用其他 source 的旧工件。
4. 要求 0 errors、0 warnings。
5. 记录 IDE/compiler 版本、构建时间、机器/环境。
6. 保存 source、project、MAP、BIN、HEX 和 build log。

本规范不把“IDE 状态栏成功”视为充分证据；必须审计实际生成工件。

## 6. 工件清单与命名

每个 flash candidate 保存：

| 工件 | 必需内容 |
|---|---|
| `main.asm` 及 include | 实际构建源码，hash |
| `project.hkproj` | chip/settings/source path 快照 |
| `main.map` | symbols、code size、table/sender page |
| `main.bin` | 实际烧录镜像，size/hash |
| `main.hex` | 独立文本工件审计，size/hash |
| build log | 0 warning 证据、工具版本 |
| board profile | pin/electrical/clock/OPTION |
| acceptance log | 烧录器、serial、结果、实板现象 |

推荐 manifest：

```json
{
  "build_id": "product-board-rev-chip-toolchain-timestamp",
  "chip": "HK64S8101",
  "toolchain": "company_ide",
  "source_sha256": "...",
  "map_sha256": "...",
  "bin_sha256": "...",
  "hex_sha256": "...",
  "bin_size_bytes": 1532,
  "highest_word": "0x02FD",
  "warnings": 0,
  "db_bytes": 512,
  "table_pairs": [
    {"table": "AVATAR_DATA0", "sender": "SEND_CHUNK0", "page": "0x00"}
  ]
}
```

## 7. MAP 审计

MAP 必查：

- reset/interrupt vector labels。
- INIT/main/subroutine 地址。
- 所有 table 和 sender 地址。
- 最高 symbol/occupied word 不越过 `0x03FF`。
- table/sender 在同一 256-word page。
- ORG 计划与实际地址一致。
- 没有符号意外落入保留区或覆盖区。

MAP 的“代码大小”字段可能含空洞或受生成器统计方式影响，必须与最高地址和 BIN size 交叉验证。

## 8. BIN 与 HEX 审计

### 普通机器字

BIN 以 16-bit word little-endian 输出。代表机器字 `0x7201` 在 BIN 中通常表现为：

```text
01 72
```

### DB

DB 物理编码按 [04-程序布局-ORG-查表规范.md](04-程序布局-ORG-查表规范.md) 的 E2 模型审计，但源码运行时仍保持原始 byte order。

### 审计项

- BIN size 为偶数，且不超过 2048 bytes。
- `bin_size == (highest_written_word + 1) * 2`，若工具定义不同则记录解释。
- reset vector 非旧工件。
- 每个 DB block marker 出现在预期 physical offset。
- HEX address 与 MAP/BIN 对应。
- 文件修改时间不能作为唯一的新鲜度证据；使用 hash 和 build ID。

## 9. 防止旧工件串用

烧录前必须建立 source-to-artifact chain：

```text
source hash
+ project settings hash
+ compiler version
+ build log
=> MAP/BIN/HEX hashes
=> flash session id
=> readback/acceptance evidence
```

禁止：

- 构建 A.asm 后烧录 state 中残留的 B.bin。
- 修改源码但未重新构建，直接沿用旧 BIN。
- 仅凭文件名相同认定工件匹配。
- 在多个 workspace 共享一个模糊的 build 目录而不保存 manifest。

## 10. 烧录前门禁

执行 [checklists/pre-flash.md](checklists/pre-flash.md)：

- flash candidate 状态成立。
- 目标板、chip、revision、serial/工位已确认。
- VDD、连接方向、烧录接口和保护措施正确。
- BIN hash 与审批 manifest 一致。
- OPTION 由独立流程确认；若未覆盖，记录为阻断或已批准的限制。
- 烧录器/COM port 唯一且可追溯。
- 有回滚/恢复镜像和断电处置。

## 11. 在线烧录

当前 CLI 的在线烧录示例：

```powershell
python D:/hk64s8x-cli/flash/scripts/flash_run.py online-program `
  --workspace D:/project `
  --port COM7 `
  --verify `
  --json
```

烧录器默认从 `state.last_build.bin_file` 取文件，因此烧录前必须打印并核对该绝对路径、size 和 hash。对 company IDE 工件，优先显式建立受审计 BIN 到烧录 session 的关联，不能盲信旧 state。

## 12. verify 的正确解释

MTP verify 成功最多说明：

- 烧录通信完成。
- 目标可读区域与发送镜像在 verify 算法下匹配。

它不证明：

- CPU 从正确 vector 启动。
- OPTION、clock、WDT 正确。
- GPIO 电气安全。
- I2C 有 ACK。
- OLED byte order/page format 正确。
- 数码管位序和极性正确。

所以 `HK-BUILD-007` 要求烧录后继续做功能验收。

## 13. 实板验收

执行 [checklists/hardware-acceptance.md](checklists/hardware-acceptance.md)，分三层：

### 通用启动层

- 供电电流、复位、clock、WDT。
- 安全 GPIO 默认态。
- heartbeat/最小可观察输出。

### 外设协议层

- I2C START/address/ACK/STOP 波形。
- GPIO 输入输出电平与极性。
- 数码管逐 COM/逐段 probe。

### 业务功能层

- OLED 分层图案直至目标图片。
- 数码管非对称 `1234`、全部数字和进位边界。
- 长时间运行、掉电重启、异常恢复。

每项记录 expected、actual、measurement、pass/fail、证据文件。

## 14. 版本变化后的重验证

以下变化触发 `HK-BUILD-008`：

- company IDE / compiler source 更新。
- instruction/register metadata 更新。
- output generator / simulator / flasher 更新。
- chip model/revision、board revision、clock、OPTION 变化。

最低重跑集合：

1. 65 instruction representative probes。
2. JMP/CALL label、EQU 和数字 target probes。
3. DB layout marker probe。
4. DB CLI omission negative probe。
5. TABL/TABH page 0 与跨页实板 probe。
6. I2C ACK release waveform。
7. 当前 board 的 OLED/数码管 known-good baseline。

## 15. 发布归档

hardware_verified 包建议只读归档：

```text
release/
  manifest.json
  source/
  project/
  artifacts/
  logs/
  board-profile.json
  measurements/
  photos-or-captures/
  deviations.json
```

文件名包含 `verified` 时，后续修改应复制成新 revision，而不是覆盖旧基线。

## 16. 最终签核

- [ ] 静态检查无 BLOCKER/ERROR。
- [ ] 目标工具链声明正确；DB 项目使用 company IDE。
- [ ] build 0 errors / 0 warnings。
- [ ] source/project/MAP/BIN/HEX/log hashes 已保存。
- [ ] program range、ORG、holes、table pages 审计通过。
- [ ] DB marker 和 byte count 审计通过。
- [ ] 烧录目标、BIN 路径、size、hash 二次核对。
- [ ] verify 结果与功能验收分开记录。
- [ ] 实板 checklist 全部通过或有正式 deviation。
- [ ] 发布包可从 manifest 复现和追溯。
