# 08 踩坑案例与症状诊断手册

## 1. 使用方法

本手册按“症状 -> 高概率原因 -> 证据 -> 检查 -> 正确修复 -> 禁止误修”组织。诊断原则：

1. 先固定 source、toolchain、artifact hash 和 board profile。
2. 每次只改变一个变量。
3. 从最小非对称 probe 开始，不直接在大图片/完整业务中试错。
4. 冲突时按 E1 > E2 > E3 > E4 > simulator > 注释/推断。
5. 保留失败 probe 和结论，但禁止把失败文件复制为正式模板。

## 2. 编译成功，但 DB 数据根本没有进入 BIN

### 症状

- Python CLI 返回 `0 error(s)`。
- 源码有 `DB`，但 BIN 小得异常。
- MAP/label 可能存在，实际数据区没有对应 bytes。
- 烧录后 OLED 字库/图片为空或读取到其他指令。

### 根因

当前 Python source module assembler 第一遍没有按 `DB` word 数推进地址，第二遍没有生成 DB machine words。`asmc_compile.py` 直接复用该模块。

### 证据

- `HK_ASM_Compiler/src/core/assembler.py:167-239`，E3。
- `.embeddedskills/build/probe_db_table.bin` 仅 46 bytes，E4。

### 正确处置

- 触发 `HK-TOOLCHAIN-DB-001`。
- 用 `builtin_compiler` 构建含 DB 的编译 release；只有用户明确要求时再用 company IDE 交叉验证或生成正式工件。
- 审计 MAP/BIN/HEX、marker 和 hash。

### 禁止误修

- 不要删除 DB 改成大量 `MOV A,#...` 只为绕过工具缺陷，除非产品设计明确接受容量/性能变化。
- 不要因为 status=success 继续烧录。

## 3. HxD 中 DB 顺序“怪异”，于是对源码做 nibble swap

### 症状

- BIN 中 `DB 12H,34H` 看起来是 `34 21`。
- 人为交换半字节或 B0/B1 后，字母轮廓/头像细节乱码。

### 根因

把 E2 物理存储排列错误地当成 E1 运行时消费顺序。公司 IDE DB 有特殊 word encoding，但实板 `TABL -> TABH` 对原始 DB 顺序是正确的。

### 正确处置

```asm
DB B0,B1
MOV A,index
TABL
CALL SEND
MOV A,index
TABH
CALL SEND
```

### 禁止误修

- 不做 nibble swap。
- 不交换 B0/B1。
- 不以 `hello_db_tabl_tabh_hxd_probe.asm` 为模板。

## 4. 小表正常，大图片第 2/3 块乱码

### 症状

- page 0 字符/图片正确。
- 表跨到 `0x01xx` 或 `0x02xx` 后错误。
- simulator 可能仍显示通过或总是读取同一块。

### 根因

`TABL/TABH` 与目标表不在同一 256-word page；或使用了一个中央 sender 跨页读取。当前 simulator 还固定读取 page 0，无法暴露真实问题。

### 正确处置

- 每个 DB block 旁放同页 sender。
- 从 MAP 检查 table/sender 和具体 TABL/TABH 地址高 8 位相同。
- 64×64 / 512-byte 可参考已验证 `224+224+64` 分块，但必须重新做容量计算。

### 禁止误修

- 不先怀疑 OLED 硬件。
- 不增加随机 NOP。
- 不以 simulator 跨页结果作为验收。

## 5. `TABL` 正确，`TABH` 每隔一个字节错误

### 症状

- 成对数据第一个 byte 对，第二个 byte 不稳定或重复。

### 根因

`TABL` 已覆盖 A；执行 `TABH` 前没有重新装载 table index。

### 正确处置

```asm
MOV A,88H
TABL
CALL I2C_SEND
MOV A,88H             ; 必须存在
TABH
CALL I2C_SEND
```

## 6. JMP/CALL 数字地址报操作数类型错误

### 症状

```text
期望操作数格式: [K], 实际: [R]
```

触发写法包括 `JMP 03FFH`、`CALL 0x03FF`。

### 根因

当前 lexer/parser 对直接数字 target 的 token 匹配问题。

### 正确处置

使用 label 或 `EQU`：

```asm
LAST_WORD EQU 03FFH
JMP LAST_WORD
```

## 7. 编译只有 warning，但程序跳到了错误地址

### 症状

- 目标超过 `0x03FF`。
- 编译器提示 warning，仍输出 BIN。
- 运行跳到低 10-bit 对应地址。

### 根因

jump/call 编码路径可能通过 `& 0x3FF` 截断超范围目标。

### 正确处置

把所有 range/truncation warning 提升为 BLOCKER，修正布局后重新构建。

## 8. ORG 让 BIN 突然变大

### 症状

- 指令很少，BIN 接近 2 KB。
- HxD 中出现大段 `00 00`。

### 根因

`ORG` 是 word address；输出按最高写入地址建立镜像，空洞填 `0x0000`。

### 正确处置

- 删除无必要 ORG。
- 报告 highest word、hole words、image bytes。
- 保留向量和查表分页所需的受控 ORG。

## 9. 不同段覆盖，但编译仍生成工件

### 症状

- 两个 label/MAP 地址重合。
- 后定义的机器字覆盖前定义内容。
- 功能表现与源码局部阅读不一致。

### 根因

工具链将 overlap 降为 warning 或采用后值覆盖。

### 正确处置

任何 occupied word 被写两次都触发 `HK-LAYOUT-004` BLOCKER。修正 ORG/块大小，禁止接受工件。

## 10. `MOV A,#0x80H` 解析失败

### 根因

同时使用 `0x` 前缀和 `H` 后缀。

### 正确处置

二选一：`#0x80` 或 `#80H`。

## 11. 公司 IDE 不识别 Python 内部 SFR 名

### 症状

- `PA_PU`、`PB_OE` 在某工具链可用，在 company IDE 源码中失败。

### 根因

它们是 Python 模块内部 alias，不是 `REG825.INC` 正式 API。

### 正确处置

源码使用 `PA_PPU`、`PB_POE`；兼容转换留在 CLI adapter。

### 2026-07 更新

新版 register metadata 已把 8 个 GPIO 名修正为正式形式，但 Python source module 的内部 alias 仍然存在，因此本规则不撤销。另有 `LVD/LVD1/LVD2/LVD3` 的 INC/metadata 差异，不能按同一套 alias 逻辑自行补全。

## 12. 用 SFR 二次 EQU 后某些指令解析异常

### 症状

- `MOV` 可能通过，`BSET/BCLR/RLR/DECSZR` 等对 alias 行为不一致。

### 根因

历史工具链的 operand/symbol resolution 对别名类别不稳定；二次 alias 还掩盖空间类型。

### 正确处置

交付源码直接使用正式 SFR 名。通过注释和 board profile 表达角色。

## 13. 数码管 `1234` 显示为 `3412`

### 根因

假设视觉顺序是 COM0,COM1,COM2,COM3；当前板实测是 COM2,COM3,COM0,COM1。

### 正确处置

修改视觉索引到 COM 的映射，不先改数字数组或段码表。

## 14. 数码管全亮、串影或多位同时亮

### 高概率原因

- 把四位都当成共阴或共阳。
- 全关值错误；当前板必须 `PA_PIO=60H`。
- 在 COM 打开时切换段码。
- 上电先开 `POE`，但 `PIO` 尚未预装安全值。

### 正确处置

逐 COM 验证极性，并固定：

```text
all off -> write segments -> enable one COM -> delay -> all off
```

## 15. 数码管 `0000 -> 0101 -> 0202`

### 高概率原因

1. BCD persistent state 与 delay/table/conversion scratch 复用。
2. 段码转换连续 `SE` 前未重新装载 A。
3. scan 和 counter 并发覆盖共享 SRAM。

### 正确处置

- 审计 `80H..BFH` 全部地址与调用链。
- 把 `90H..93H` 等 persistent 区和 scratch 分开。
- 每次 `SE` 前 `MOV A,saved`。
- 在 frame 边界更新显示快照。

## 16. I2C 没有 ACK 或偶发花屏

### 高概率原因

- 发送 8 位后仍主动驱动 SDA high，与从机 ACK low 争用。
- 7-bit `0x3C` 与 wire write byte `0x78` 混淆。
- 上拉、电压或 pin mapping 错。
- NOP 时序与实际 clock 不匹配。
- NACK 被忽略，事务仍继续。

### 正确处置

第 9 个时钟前：

```asm
BCLR PB_POE,7
BSET PB_PIO,6
MOV A,PB_PIO
AND A,#80H
BCLR PB_PIO,6
BSET PB_POE,7
BSET PB_PIO,7
```

再用逻辑分析仪确认 SDA driver 释放和真实 ACK。

## 17. OLED 执行 `A5H/AFH` 仍看不到目标图

### 根因

强制全亮命令不能证明 GDDRAM data path、control byte、window、byte count 和图片格式。

### 正确处置

按阶梯：强制全亮 -> GDDRAM 1024×FF -> 00/FF 条纹 -> 非对称 marker -> 字符 -> 图片。

## 18. OLED 图像横竖/上下错位

### 高概率原因

- 把每 byte 当横向 8 pixels，而 SSD1306 page format 是纵向 8 pixels。
- bit order 反了；当前规则 bit0 是 page 顶部。
- window 宽高与发送 byte count 不一致。
- remap/scan direction 与资产转换重复翻转。

### 正确处置

保存资产转换参数；用单列 bit probe `01,02,04,08,10,20,40,80` 确认方向。

## 19. simulator 通过，实板失败

### 根因类别

- simulator `TABL/TABH` 固定 page 0。
- 电气层（上拉、争用、电平、时序）未建模。
- OPTION、clock、复位、外设状态未完整建模。

### 正确处置

将 simulator 限定为低等级辅助证据。冲突时服从 E1/E2，并创建最小实板 probe。

## 20. 烧录 verify 成功，功能仍错误

### 根因

verify 只比较程序存储，不覆盖源码意图、工具缺陷、OPTION、外设协议和业务逻辑。

### 正确处置

核对 source->artifact hash 链，并执行板级功能 checklist。不得把 verify success 写成 hardware_verified。

## 21. 旧 BIN 被误烧录

### 症状

- 源码已修改，板子行为完全不变。
- state 指向另一个 workspace 的 BIN。
- build 目录时间戳/文件名容易混淆。

### 正确处置

烧录前打印 source hash、BIN absolute path、size/hash、build ID；保存到 flash session。禁止只看文件名。

## 22. AI “优化” verified 大表后回归

### 根因

AI 格式化、去重、换行重排、nibble swap 或按 HxD 调整了已验证 DB，但没有做 byte count/hash 回归。

### 正确处置

- `verified` 文件默认只读。
- 复制成新实验 revision。
- 对 DB 修改输出 before/after byte count、hash、首尾 marker 和视觉回归。

## 23. 受限指令被误当成常规能力

| 指令 | 风险 | 处置 |
|---|---|---|
| `RET A,#K` | 只有编码证据，无硬件语义证据 | 正式项目禁用 |
| `CPL/CPLR` | 语义/写回目标说明有疑点 | 关键逻辑禁用，先做真值表 |
| `IDLE/STOP` | 唤醒、WDT、GPIO 保持未知 | 完成 board power-mode test 后采用 |

## 24. LED 常亮、只变化一次或闪烁节拍异常

### 症状

- LED 初始化后常亮或常灭，延时循环似乎没有结束。
- 外层计数寄存器值不变，但 A 在变化。
- 加入 `CLRWDT` 后不再复位，却永久卡在循环中。

### 根因

`DECSZ R` 的结果写入 A，不写回 R；`INCSZ R` 同理。若把它们直接接向后跳转并期望 R 持久递减/递增，每轮都会从同一个 R 重新计算，循环无法按预期进展。`CLRWDT` 只能避免看门狗复位，不能修复无进展循环。

### 正确处置

- 从 `instruction-reference.json.raw_notes` 核对每条候选指令的写回目标。
- 持久 R 计数使用 `DECSZR/INCSZR`，或显式把 A 写回 R。
- 对每条回边证明至少一个 loop-carried 状态在所有继续路径更新，并证明退出条件可达。

## 25. 目标 500 ms 实际约 4 s

### 根因

把 16 MHz OSC 直接当作指令执行时钟，忽略 `SCK_PS` 分频。HK64S825 默认 `SCK_PS=34H` 时，OSC 16 MHz 经 8 分频得到实际 SCK 2 MHz；延时循环必须用 SCK 和指令 cycle 数计算。

### 正确处置

记录 `osc_hz`、实际或 reset `SCK_PS`、派生 `sck_hz`、总 cycles、目标微秒和误差百分比。精确时序无法静态解析时失败关闭，不用“16 MHz 大概值”代替。

## 26. 快速分诊树

```text
编译失败？
  -> 先查语法、正式 SFR 名、JMP/CALL target、立即数范围
编译成功但工件异常？
  -> 查 DB 工具链、ORG holes、overlap、MAP/BIN/hash
烧录成功但总线无响应？
  -> 查 pin/电压/address/ACK release/clock
OLED 小样正常、大表错？
  -> 查 page format、byte count、TABL/TABH 同页、A 重装
数码管固定图案错？
  -> 查 COM 位序、混合极性、全关值、段码
动态状态错？
  -> 查 SRAM 生命周期、SE 副作用、并发更新
LED 卡住或只变化一次？
  -> 查 DECSZ/INCSZ 写回目标、循环进展和 CLRWDT 是否掩盖死循环
延时约放大 8 倍？
  -> 查 OSC、SCK_PS、实际 SCK 和 cycle audit
```
